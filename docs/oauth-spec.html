OAuth 2.0 Specification

Technical spec for Nothing Health API authentication. Prepared for internal alignment — March 2026.

What We Need to Build

Nothing doesn't have OAuth today. We need to build it so third-party apps can request permission to access a user's health data — with the user's explicit consent, scoped to specific data types.

Why OAuth 2.0?

It's the industry standard (used by Google, Apple Health, WHOOP, Fitbit). Users never share their Nothing password with third-party apps. They just approve specific permissions, and can revoke anytime.

Standard OAuth 2.0 Authorization Code flow. Here's the step-by-step:

Token Design

Access Token (JWT)

Short-lived, used for API calls. Contains the user ID and granted scopes.

{
  "sub": "user-uuid-here",        // Nothing account ID
  "scopes": [                     // What this token can access
    "heart_rate:read",
    "sleep:read"
  ],
  "exp": 1711540800,              // Expires in 30 minutes
  "type": "access",
  "iss": "nothing-health-api"     // Issued by us
}
PropertyValueRationale
FormatJWT (HS256)Stateless verification, no DB lookup per request
Lifetime30 minutesShort window limits damage if leaked
Contains scopesYesAPI can check permissions without DB call

Refresh Token (JWT)

Long-lived, used only to get new access tokens. Stored in our database for revocation.

{
  "sub": "user-uuid-here",
  "scopes": ["heart_rate:read", "sleep:read"],
  "exp": 1719316800,              // Expires in 90 days
  "type": "refresh",
  "jti": "unique-token-id"        // For revocation tracking
}
PropertyValueRationale
Lifetime90 daysUsers don't need to re-authorize frequently
RevocableYes (via jti)One-tap revoke from Nothing X kills all tokens
DB-trackedYesWe can revoke specific tokens instantly

Authorization Code

PropertyValue
Lifetime5 minutes
One-time useYes — marked as used after exchange
Bound to clientYes — must match the requesting client_id

Permission Scopes

Each scope grants read access to one data type. Users choose which scopes to approve on the consent screen. Apps should request only what they need.

ScopeData AccessStatus
heart_rate:readContinuous + on-demand HR readingsREADY
sleep:readSleep sessions, stages, scoresREADY
spo2:readBlood oxygen percentageREADY
steps:readDaily step count, distance, caloriesREADY
workouts:readExercise sessions with HR zonesREADY
profile:readDisplay name only — no email/phoneREADY
skin_temperature:readWrist temp + baseline deviationFUTURE
menstrual_cycle:readCycle dates, phases, ovulationFUTURE
Sensitive scopes

menstrual_cycle:read is classified as sensitive health data. Apps requesting this scope will undergo additional review and must demonstrate a clear health/wellness use case.

Consent Screen (Nothing X)

When a third-party app requests authorization, the user sees a consent screen inside the Nothing X app. This needs to be built by the NTX app team.

Required elements

  • App name and icon — from the developer portal registration
  • Requested permissions — listed individually with checkboxes (all on by default)
  • Developer name — who made the app
  • Allow / Deny buttons — clear, prominent
  • Users can uncheck individual scopes — they might approve heart rate but deny sleep

What happens on revoke

  1. All refresh tokens for that app + user are marked as revoked in the database
  2. Existing access tokens expire naturally within 30 minutes (JWT — can't revoke instantly)
  3. App gets 401 Unauthorized on next token refresh attempt
  4. User sees confirmation: "Access revoked for MyFitnessApp"

OAuth API Endpoints

All under /api/v1/oauth/.

POST /register

Developer registers their app. Returns client credentials.

Request:
{
  "name": "My Health App",
  "redirect_uri": "https://myapp.com/callback",
  "scopes": "heart_rate:read sleep:read steps:read"
}

Response:
{
  "client_id": "nth_abc123...",
  "client_secret": "nths_xyz789...",     ← show once, never again
  "name": "My Health App",
  "scopes": "heart_rate:read sleep:read steps:read"
}

POST /authorize

Initiates consent flow. In production, this triggers the Nothing X consent screen.

Request:
{
  "client_id": "nth_abc123...",
  "redirect_uri": "https://myapp.com/callback",
  "scope": "heart_rate:read sleep:read",
  "state": "random-csrf-token",
  "response_type": "code"
}

Response (redirect to):
https://myapp.com/callback?code=AUTH_CODE_HERE&state=random-csrf-token

POST /token

Exchange auth code or refresh token for access + refresh tokens.

Request (authorization code):
{
  "grant_type": "authorization_code",
  "code": "AUTH_CODE_HERE",
  "client_id": "nth_abc123...",
  "client_secret": "nths_xyz789...",
  "redirect_uri": "https://myapp.com/callback"
}

Request (refresh):
{
  "grant_type": "refresh_token",
  "refresh_token": "REFRESH_TOKEN_HERE",
  "client_id": "nth_abc123...",
  "client_secret": "nths_xyz789..."
}

Response:
{
  "access_token": "eyJ...",
  "refresh_token": "eyJ...",
  "token_type": "Bearer",
  "expires_in": 1800,
  "scope": "heart_rate:read sleep:read"
}

POST /revoke

Revoke a refresh token. Called from Nothing X or programmatically.

Request:
{
  "token": "REFRESH_TOKEN_HERE",
  "client_id": "nth_abc123...",
  "client_secret": "nths_xyz789..."
}

Response:
{ "status": "revoked" }
